# Adalight как «ядро + моды» (проект архитектуры)

Цель: оставить в ядре только **загрузчик модов и оболочку интерфейса**, а всю
содержательную функциональность — эффекты, цветомузыку, захват экрана,
транспорты, уведомления — вынести во **встроенные моды**, идущие в комплекте
(как базовый мод в Factorio). Пользовательские моды подключаются тем же
контрактом.

Решения (согласовано):
- **Дробность — мод на группу**: «Эффекты лампы», «Цветомузыка», «Захват
  экрана», «Транспорты», «Уведомления» (+ языки-локали). Несколько крупных
  встроенных модов.
- **Отключение — можно, но с предупреждением**: базовый мод разрешено
  выключить, но менеджер предупреждает, что часть функционала пропадёт.
- **Порядок — сначала спроектировать всё ядро** (этот документ), потом
  переносить по этапам.

## 1. Что остаётся ядром (не мод)

| Слой ядра | Ответственность |
|---|---|
| **Загрузчик модов** (`ModHost`, эволюция `PluginManager`) | Обнаружение встроенных и пользовательских модов, вызов `register(api)`, вкл/выкл, изоляция ошибок |
| **Оболочка GUI** | Сайдбар, страницы, статус-карточка, менеджер модов, предпросмотр. Панели **наполняются из реестров**, а не хардкодом |
| **Движок-скелет** (`engine.py`) | Цикл «источник → обработка → транспорт». Сам не знает конкретных источников/эффектов/транспортов — берёт их из реестров |
| **Конвейер цвета** (`device.py` pipeline) | Гамма/яркость/насыщенность/температура/баланс белого + расписание/адаптив/ночь. Плумбинг: обрабатывает то, что дали источник/эффект. Пока ядро (кандидат в мод позже) |
| **Событийная шина** | Публикация/подписка на события ядра |
| **Конфиг, i18n** | Хранилище настроек, слой перевода (русский — исходный) |

Всё остальное — моды.

## 2. Единый контракт мода

Мод — объект (из `create_plugin()`), необязательно с полями:

```python
class MyMod:
    name = "my_mod"            # ключ в конфиге
    title = "Мой мод"          # видно в менеджере
    description = "…"
    version = "1.0"
    base = False               # True у встроенных: предупреждать при выключении

    def register(self, api):   # регистрация возможностей в реестрах ядра
        ...
    def start(self, api, settings):  # (опц.) фоновая работа — как уведомления
        ...
    def stop(self):            # (опц.)
        ...
    settings_schema = [...]    # (опц.) авто-форма настроек
```

Ключевой сдвиг: **регистрация возможностей происходит на включении, а не при
загрузке.** Выключил мод → его эффекты/источники/транспорты уходят из реестров
(и из списков в GUI). Включил → возвращаются. Так «выключить базовый мод»
реально убирает функционал (и оправдывает предупреждение).

### API (расширение `PluginAPI`)

| Метод | Что регистрирует |
|---|---|
| `register_lamp_effect(id, label, render, wants_color=…, wants_speed=…)` | эффект лампы (**уже есть**) |
| `register_music_effect(id, label, factory, wants_color=…)` | эффект цветомузыки |
| `register_capture_source(id, label, factory, platforms=…)` | источник захвата экрана |
| `register_transport(id, label, factory, wants_serial=…, wants_network=…)` | транспорт (serial/WLED/…) |
| `on(event, handler)` / `emit(event, **data)` | событийная шина |
| `trigger_flash / notify / log` | вспышки/уведомления/лог (**уже есть**) |

`factory` для источника/транспорта — функция `(cfg) -> объект` с нужным
интерфейсом (у источника — `get_frame()/get_bands()/width/height/close()`;
у транспорта — `connect()/send_raw()/close()`). Интерфейсы фиксируем как
`Protocol`, чтобы моды не зависели от внутренностей.

## 3. Реестры ядра

Одна общая механика: `Registry[id -> Spec]`, каждый элемент помнит, **каким
модом** зарегистрирован (чтобы снять при выключении).

- `effects.lamp` — эффекты лампы (**есть**).
- `effects.music` — эффекты цветомузыки (сейчас `MusicRenderer` — класс с
  состоянием; провайдер отдаёт фабрику рендерера по id).
- `capture.sources` — источники: `auto` и движок выбирают из реестра по
  платформе и доступности (нынешняя логика `create_backend` переезжает сюда).
- `transports` — `device.py` становится тонким фасадом: транспорт берётся из
  реестра по `cfg.transport`.

GUI (комбо эффектов, «Транспорт», «Бэкенд») строится из реестров и обновляется
при вкл/выкл модов — как уже сделано для эффектов лампы.

## 4. Событийная шина

Простой pub/sub (`EventBus.subscribe(event, cb)` / `emit(event, payload)`).
События ядра: `engine.started/stopped`, `lighting.on/off`, `frame`,
`notification.received`, `update.available`. Моды подписываются, чтобы
реагировать на состояние, и эмитят свои. Уведомления-мод при этом продолжает
дергать `trigger_flash` (вспышка — не событие шины, а действие на ленте).

## 5. Включение/выключение с предупреждением

- У каждого мода `enabled` в конфиге; у встроенных `base=True`, по умолчанию
  включены.
- Выключение мода с `base=True` → диалог «Выключить «…»? Часть функционала
  может пропасть (эффекты/захват/транспорт)». Community-моды выключаются без
  предупреждения.
- Мягкая деградация ядра, если возможности нет:
  - нет ни одного эффекта лампы → в списке «нет эффектов», режим «Лампа»
    сообщает об этом;
  - нет источника захвата → режим «Захват экрана» недоступен, подсказка;
  - нет транспорта → «Старт» показывает дружелюбную ошибку «нет транспорта».

## 6. Встроенные группы-моды (в комплекте)

`adalight/mods/` (каждый — обычный мод с `create_plugin()`, `base=True`):

| Мод | Регистрирует |
|---|---|
| `effects_lamp` | solid, gradient, rainbow(+static), breathing, fire, comet, aurora, starry |
| `effects_music` | spectrum, pulse, wave, beat |
| `capture` | bettercam, dxcam, mss (Win), wf-recorder, grim (Wayland) |
| `transports` | serial (Adalight), wled |
| `notifications` | вспышки уведомлений (**уже мод**) |
| `locale_en` | английский язык (**уже через `create_locale`**) |

Все видны в менеджере как встроенные — это и закрывает «почему не видно
плагин»: эффекты станут модом «Эффекты лампы», который виден и управляем.

## 7. План миграции (по этапам, приложение работает на каждом)

- **Этап 0 (готово):** реестр эффектов лампы + `register_lamp_effect`,
  `render_lamp` через реестр, GUI из реестра.
- **Этап 1 (готово):** регистрация возможностей по `enabled` + снятие при
  выключении (`unregister_source`); базовые моды (`base=True`) включены по
  умолчанию; менеджер показывает встроенные моды и предупреждает при выключении
  базовых; встроенные эффекты лампы переехали в бандл-мод `effects_lamp` (виден
  и управляем); мягкая деградация (нет эффектов → плейсхолдер, движок не падает).
- **Этап 2 (готово):** реестр `effects.music` (фабрики рендереров, состояние
  per-сессия) + мод `effects_music` (base=True); `PluginAPI.register_music_effect`;
  цвет/чувствительность читаются на лету, смена эффекта пересоздаёт рендерер;
  мягкая деградация (мод выключен → гашение музыки).
- **Этап 3 (готово):** реестр источников захвата + мод `capture` (base=True);
  `create_backend`/`auto` читают реестр (выбор по платформе и приоритету с
  фолбэком); `PluginAPI.register_capture_source`; CLI поднимает базовые моды
  перед движком (иначе headless остался бы без источников/эффектов); мягкая
  деградация (мод выключен → «Захват экрана» без источников).
- **Этап 4 — транспорты:** реестр `transports` + мод `transports`; `device.py`
  тонкий фасад.
- **Этап 5 — событийная шина:** `EventBus`, перевод уведомлений/тостов на неё.

Каждый этап — отдельный релиз, тесты зелёные, обратная совместимость.

## 8. Совместимость и риски

- Старые пользовательские моды (`create_plugin`/`create_locale`) работают без
  изменений.
- Конфиг хранит id эффекта/источника/транспорта; валидация мягкая (по реестру,
  как уже сделано для эффектов).
- Порядок загрузки: встроенные моды регистрируются **до** старта движка.
- Главный риск — выключенный базовый мод оставляет без функционала; закрываем
  предупреждением и мягкой деградацией.

## 9. Пока оставляем в ядре

Конвейер цвета и планировщик яркости (гамма/яркость/расписание/адаптив) — это
плумбинг, через который проходят кадры от любого источника/эффекта. Вынос его
в мод трогает сам движок сильнее всего; предлагаю оставить в ядре на этих
этапах и вернуться к нему последним, когда контракты источников/эффектов/
транспортов устоятся.
